## Reported-only (NOT applied — risk assessment)

### R1 — `dragSession.ts` is a write-only registry (published, never consumed) — MEDIUM risk
`packages/studio/src/utils/dragSession.ts`. `beginDragSession` / `endDragSession` are called live from 3 sidebar files (`BlocksTab.tsx`, `AudioRow.tsx`, `AssetsTab.tsx`), but the only reader — `getActiveDragSession` — has **no non-test consumer** (only `dragSession.test.ts` reads it, purely as the assertion vehicle for begin/end). So the module's `active` state is currently written on every in-studio drag and never read: the "in-window drag payload registry" that its own doc comment describes (drag-over ghost/snap knowing what's being dragged) is **not actually wired to anything**. Either the drag-over UI consumer was never landed, or it was removed in a pivot.
- **Not applied because:** three safe-looking options each have a catch. (a) Delete `getActiveDragSession` only → guts the test (it's the sole observation point) and leaves `active` still write-only. (b) Delete the whole module → touches 3 live files and *might* be removing a half-finished feature the author intends to complete. (c) Wire a real consumer → new behavior, out of audit scope. This needs an author decision on intent, not a mechanical fix. fallow does *not* flag it (test file counts as a consumer in its import graph), so it is not a gate blocker.

### R2 — Clamp helper duplication — LOW risk / LOW value
Four `Math.min(Math.max(...))` clamps: canonical `clampNumber(value,min,max)` at `packages/studio/src/utils/studioHelpers.ts:226`; domain copies `clampPreviewZoomPercent` (`components/nle/previewZoom.ts`), `clampTimelineZoomPercent` (`player/components/timelineZoom.ts`), `clampPanelNumber` (`components/editor/propertyPanelHelpers.ts`).
- **Not applied because:** the three domain clamps carry their own named min/max constants and read as intent-revealing at their call sites; folding them into `clampNumber(v, MIN, MAX)` is a lateral move that changes each helper's signature/exports and the tests that pin them, for ~0 LOC saved. Behavior-adjacent churn against a "keep it working" constraint with no real payoff. Leave as-is.

### R3 — Scattered pixel↔time conversions — LOW risk / MEDIUM value
Inline `/ Math.max(pixelsPerSecond, 1)` and `* pixelsPerSecond` recur in `player/components/timelineEditing.ts:112,183`, `captions/components/CaptionTimeline.tsx:75,115,141-142`, alongside the canonical `getTimelinePlayheadLeft` (`player/components/timelineLayout.ts:218`). Extracting `pixelToTime()` / `timeToPixel()` helpers would de-duplicate ~6 sites.
- **Not applied because:** the inline forms differ subtly (scroll-delta terms, gutter offsets, `Math.max(...,1)` guards); a shared helper risks quietly changing edge behavior at drag boundaries. This is a real, worthwhile refactor but it is behavior-adjacent and belongs in its own reviewed change, not a dead-code sweep.

### R4 — `furthestClipEnd` computed three ways — LOW risk / LOW value
`furthestClipEndFromDocument` (`player/lib/timelineElementHelpers.ts:47`) is canonical; `App.tsx:101-105` and `blockInstaller.ts:136` each inline the same `max(start+duration)` reduce. They legitimately read different sources (live DOM vs in-memory arrays), so they are not literal duplicates — only the reduce shape repeats. A generic `maxEndFromClips(clips)` could absorb the two inline reduces. Low value, defer.
