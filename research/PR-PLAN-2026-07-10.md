## TASK 2 — Stack design

Stacked branches off `origin/main`, each based on the previous. Per-PR cherry-pick is not viable (136 interleaved commits sharing glue files); the stack is carved by **path-scoped checkout from the rebased tip** (see Task 4). Dependency order (bottom → top), each independently green (`bun run build` + `tsc --noEmit` + suite):

```
origin/main
   └─ PR1  timeline DnD core            (base of everything; zero upward deps)
        └─ PR2  NLE shell + soft reload  (+ asset-preview overlay half)
             └─ PR3  visual refresh + thumbnails
                  └─ PR4  canvas element editor
                       └─ PR5  assets sidebar
```

Import-direction verified: timeline-core imports nothing from the layers above it; `PreviewPane`→`AssetPreviewOverlay` is the one cross-area edge and is satisfied by putting the overlay in PR2 (below the sidebar in PR5).

**Execution model for shared files:** each changed file belongs to **exactly one PR** (its owner = the earliest PR in dependency order that needs any of its changes) and ships at its final branch-tip content there. Later PRs inherit it unchanged. This sidesteps per-hunk apportioning entirely; the only cost is that a glue file's changes are reviewed in one PR even when they serve several features (called out per file in Task 4).

**Answer to the team's question:** fallow complexity/duplication findings are **BLOCKING**, not advisory — the CI `Fallow audit` job runs the exact `--fail-on-issues` command and fails the build (it also posts a sticky PR comment, but the job itself gates). The 600-line filesize limit is the opposite: **advisory in CI** (no job), enforced only by the local lefthook hook — so it will stop your commits but won't independently red-X a PR.
