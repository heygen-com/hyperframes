## TASK 2 — CI-gate health

### (a) Build gate — FIXED via dependency-only work; `bun run build` now passes (exit 0)

**Diagnosis.** The reported symptoms (missing `tsup` for parsers/studio-server/lint; two `puppeteer-core` versions — 25.3.0 + 24.43.1 — clashing in `producer/probeStage.ts`) were **not** a dependency-file problem. All seven `packages/*/package.json` pin `puppeteer`/`puppeteer-core` uniformly at `^25.2.1`, and `bun.lock` resolves a **single** version (`25.3.0`) with **zero** `24.43` references. The `24.43.1` copy was a stale entry left behind in bun’s isolated store (`node_modules/.bun/puppeteer-core@24.43.1/`), and `tsup` simply hadn’t been materialized. Plain `bun install` reported "no changes" (it considered the degraded store satisfied); `bun install --force` also left the stale `.bun` dirs in place.

**Fix (dependency files untouched):**

```
rm -rf node_modules && bun install     # 1557 packages, clean
```

After the clean reinstall: `node_modules/.bun/` contains **only** `puppeteer-core@25.3.0` / `puppeteer@25.3.0` (the 24.43.1 pair is gone), and `bun run build` completes **end-to-end (exit 0)** including `@hyperframes/producer` and `@hyperframes/cli`. `bun.lock` and every `package.json` are **unchanged** (`git status` clean for both). **No pin was needed; no source or dependency-file edit was required.**

> Note: `node_modules/.bin/tsup` does not exist at the repo root even when healthy — bun’s isolated linker keeps per-package binaries under `.bun/`; each package’s `tsup` build resolves correctly. Its absence at root is not the bug.

### Repo conventions (from `.github/pull_request_template.md` + `CONTRIBUTING.md` + `docs/contributing.mdx`)

- PR template sections: **What / Why / How / Test plan** (with a 3-item checklist: unit tests, manual testing, docs).

- "**Keep PRs focused — one feature or fix per PR.**" ≥1 maintainer approval.
- Alpha-only work targets `next`, not `main`.

### Reality check: cherry-pick vs. stacked rebase

The 103 commits are **heavily interleaved** — a single history walk shows visual-restyle, thumbnail, canvas-context-menu, and timeline-collision commits alternating (e.g. `6d3d238bc` visual restyle → `26d43ef79` canvas menu → `bb328144b` collision-push → `7cadebf99` full-width layout, all adjacent). A clean per-PR `cherry-pick` is **not realistic**: the four feature areas share glue files (`App.tsx`, `Timeline.tsx`, `useTimelineEditing.ts`, `playerStore.ts`, `TimelineCanvas.tsx`, `TimelineToolbar.tsx`) that nearly every commit touches, so cherry-picks would collide constantly.

**Recommended path: a stacked sequence off one rebased base.**

1. First **rebase the whole branch onto `origin/main`** and resolve the 15 conflicts once (adopting main’s lane-model + selection-single-sourcing as the base per Task 1). Do this on a throwaway integration branch — **not** in this worktree (the audit agent owns `packages/studio/src`).
2. From the rebased tip, build **stacked branches** by `git checkout -p` / path-scoped resets, each depending on the previous, in the dependency order below. Each layer is a PR whose base is the previous PR’s branch (GitHub stacked PRs), so reviewers see a clean incremental diff even though history was interleaved.
3. Net-new files (no `origin/main` counterpart — 36 of them) move wholesale into their owning PR with zero conflict risk; only the ~6 shared glue files need careful per-PR apportioning.

This trades one hard rebase for four clean, independently reviewable diffs — far cheaper than 103 conflict-prone cherry-picks.

### Dependency order (bottom → top)

```
PR0 (docs, optional/last) ── research/ + plan docs, oxfmt-clean
                    │
PR1  timeline-DnD core  ◄── foundation: drop model / zones / collision / drag-commit / multi-select+marquee / move+resize
```

PR1 must land first (everything references its timeline model). PR4 (canvas overlay editor) is the most independent and could arguably go earlier, but its `App.tsx`/shell touchpoints are cleaner once PR2’s `EditorShell` exists, so it stacks last.
